# Phase 7: Guided Review + Compression Coach - Context

**Gathered:** 2026-03-02
**Status:** Ready for planning
**Source:** User design discussion (chat1.txt)

<domain>
## Phase Boundary

This phase delivers the full AI-guided GTD weekly review experience: a structured three-phase flow (Get Clear, Get Current, Get Creative) using ConversationTurnCard question flows, a compression coach that identifies stale/redundant atoms with AI-written reasoning, and a draft atom staging area where AI-proposed changes await explicit user approval before being written.

</domain>

<decisions>
## Implementation Decisions

### GTD Review Flow Structure
- Three phases: Get Clear → Get Current → Get Creative
- Each phase is convergent or bounded — not open-ended chat
- Get Clear: inbox zero processing (process all inbox items)
- Get Current: review active projects, check for stale items, ensure every project has a next action
- Get Creative: data-driven, bounded, capture-oriented — NOT a chatbot

### Get Creative Design (locked — user's primary concern)
- Someday/Maybe scan (3-5 items, AI-prioritized)
- Area gap check (one question per area — "Any new projects needed?")
- Trigger list (5-8 GTD-style prompts: Health, Career, Family, Home, etc.)
- Pattern surfacing (AI-generated, based on recent decisions/insights data)
- "Anything else?" — one final freeform capture, then done
- Each step is a ConversationTurnCard: 3-4 options + freeform
- Output is atoms (captured immediately), not chat history
- Fixed progression through steps, no back-and-forth conversation
- 5-8 steps total, then phase is complete

### Context Management for Long Reviews (locked)
- Summarize at phase boundaries, not after every step
- Each API call gets: system prompt + phase summaries (~200 tokens) + current step data + user response
- Summaries generated by AI (one extra API call per phase transition)
- Keeps every API call under ~2000 tokens regardless of review length
- Summaries stored in ReviewPhaseContext (in-memory during review, persisted to session storage)
- Final review summary saved as analysis atom

### Compression Coach Scope (locked)
- One engine, two entry points
- Entry point 1: Integrated into Get Current phase of the weekly review (deep mode)
- Entry point 2: Standalone from orb (quick mode, on-demand) — natural extension
- Phase 7 builds the engine + review integration; orb entry point is thin wrapper
- Compression candidates include AI-written explanations referencing specific signals (staleness, link density, similar atoms, relevant decisions)

### Staging Area Design (locked)
- Holds three types: new atom proposals, mutation proposals, deletion/archive proposals
- Staging is ephemeral — proposals exist only during the review session
- Unapproved proposals discarded when session ends (no accumulating "pending AI suggestions")
- Staging is a UI concept, not a data concept — proposals are in-memory objects, become real atoms only when approved
- Batch display with individual approve/reject — "Approve all" as convenience, never default
- Integrates with review flow: compression candidates and next-action suggestions go to staging during Get Current; new project/task suggestions go to staging during Get Creative
- End of review: "You have N pending suggestions. Review now?"
- Every approved change goes through normal mutation pipeline with source: 'ai'
- Changelog records all AI mutations; existing undo system reverses completely

### Cloud API Escalation
- Get Clear and Get Current: can use local LLM (WebLLM) for classification tasks
- Get Creative: escalates to cloud API for reasoning (pattern surfacing, area gap analysis)
- Compression explanations: cloud API for nuanced reasoning about why atoms are stale

</decisions>

<specifics>
## Specific Ideas

- ReviewPhaseContext interface should include: phaseSummaries (string[]), currentPhase ('get-clear' | 'get-current' | 'get-creative'), currentStep (number), atomsReviewed (string[] of IDs), actionsTaken (ReviewAction[])
- Get Creative's "area gap check" should flag areas with no active projects and surface patterns like "You haven't captured anything about finances in 3 weeks"
- Compression candidates during Get Current should show: "This task has been stale for 45 days. Here's why it might be noise: [reasoning]" with options to archive, delete, reactivate, or skip
- The staging area shows: proposed atom with type/content/section/tags for new atoms; current atom + proposed change + reasoning for mutations; atom + reasoning + proposed action for deletions

</specifics>
