## User Constraints (from CONTEXT.md)

### Locked Decisions

**GTD Review Flow Structure**
- Three phases: Get Clear → Get Current → Get Creative
- Each phase is convergent or bounded — not open-ended chat
- Get Clear: inbox zero processing (process all inbox items)
- Get Current: review active projects, check for stale items, ensure every project has a next action
- Get Creative: data-driven, bounded, capture-oriented — NOT a chatbot

**Get Creative Design (locked — user's primary concern)**
- Someday/Maybe scan (3-5 items, AI-prioritized)
- Area gap check (one question per area — "Any new projects needed?")
- Trigger list (5-8 GTD-style prompts: Health, Career, Family, Home, etc.)
- Pattern surfacing (AI-generated, based on recent decisions/insights data)
- "Anything else?" — one final freeform capture, then done
- Each step is a ConversationTurnCard: 3-4 options + freeform
- Output is atoms (captured immediately), not chat history
- Fixed progression through steps, no back-and-forth conversation
- 5-8 steps total, then phase is complete

**Context Management for Long Reviews (locked)**
- Summarize at phase boundaries, not after every step
- Each API call gets: system prompt + phase summaries (~200 tokens) + current step data + user response
- Summaries generated by AI (one extra API call per phase transition)
- Keeps every API call under ~2000 tokens regardless of review length
- Summaries stored in ReviewPhaseContext (in-memory during review, persisted to session storage)
- Final review summary saved as analysis atom

**Compression Coach Scope (locked)**
- One engine, two entry points
- Entry point 1: Integrated into Get Current phase of the weekly review (deep mode)
- Entry point 2: Standalone from orb (quick mode, on-demand) — natural extension
- Phase 7 builds the engine + review integration; orb entry point is thin wrapper
- Compression candidates include AI-written explanations referencing specific signals (staleness, link density, similar atoms, relevant decisions)

**Staging Area Design (locked)**
- Holds three types: new atom proposals, mutation proposals, deletion/archive proposals
- Staging is ephemeral — proposals exist only during the review session
- Unapproved proposals discarded when session ends (no accumulating "pending AI suggestions")
- Staging is a UI concept, not a data concept — proposals are in-memory objects, become real atoms only when approved
- Batch display with individual approve/reject — "Approve all" as convenience, never default
- Integrates with review flow: compression candidates and next-action suggestions go to staging during Get Current; new project/task suggestions go to staging during Get Creative
- End of review: "You have N pending suggestions. Review now?"
- Every approved change goes through normal mutation pipeline with source: 'ai'
- Changelog records all AI mutations; existing undo system reverses completely

**Cloud API Escalation**
- Get Clear and Get Current: can use local LLM (WebLLM) for classification tasks
- Get Creative: escalates to cloud API for reasoning (pattern surfacing, area gap analysis)
- Compression explanations: cloud API for nuanced reasoning about why atoms are stale

## Summary

Phase 7 is the most architecturally complex phase of the project. It introduces three interconnected systems: a GTD weekly review state machine (three bounded phases, each with its own step queue and LLM-generated questions), a compression coaching engine (AI-written per-candidate reasoning referencing staleness, link density, and semantic similarity signals from the store), and a staging area (ephemeral in-memory proposal queue where AI changes wait for explicit user approval before touching Dexie).

The good news is that all the foundational infrastructure is already in place. The mutation pipeline with `source: 'ai'` tracking exists in `changelog.ts`. The undo system already works for any mutation by reading `before` snapshots. The AIQuestionFlow component pattern (options + freeform, module-level signals) is already established. The WebLLM browser adapter supports structured JSON via XGrammar. The cloud adapter exists. The review session persistence mechanism exists. This phase primarily assembles these pieces into higher-level orchestration rather than introducing new primitives.

The key architectural insight is that the three GTD phases map cleanly to three existing system concerns: Get Clear = inbox processing (existing triage infrastructure), Get Current = stale/project scanning (existing briefing analysis infrastructure + new compression coach), Get Creative = AI-assisted capture (new cloud API reasoning). The staging area is the connective tissue — proposals generated during all three phases accumulate there and are batch-reviewed before anything commits.

**Primary recommendation:** Build in two parallel tracks — (1) the review flow orchestration (state machine, phase transitions, step sequencing, context summarization), and (2) the staging area + mutation pipeline. These can be developed and tested independently before wiring together in the final integration plan.

### Pattern 1: Review Flow State Machine
**What:** A three-phase state machine living in `src/ai/review-flow.ts`. Each phase generates a `ReviewFlowStep[]` queue that the UI consumes one step at a time. Phase transitions trigger context summarization before the next phase begins.
**When to use:** For any multi-phase AI-guided workflow that must stay under token budget.

```typescript
// src/ai/review-flow.ts — new module
export type ReviewPhase = 'get-clear' | 'get-current' | 'get-creative';

export interface ReviewPhaseContext {
  phase: ReviewPhase;
  phaseSummaries: string[];           // AI-generated summary per completed phase (~200 tokens each)
  currentStep: number;
  atomsReviewed: string[];            // atom IDs processed during this session
  actionsTaken: ReviewAction[];       // what user did at each step
}

export interface ReviewFlowStep {
  stepId: string;
  phase: ReviewPhase;
  question: string;
  options: ReviewStepOption[];
  allowFreeform: boolean;
  context?: string;                   // atom title, project name, etc. shown above question
  atomId?: string;                    // for steps tied to a specific atom
}

export interface ReviewStepOption {
  id: string;
  label: string;
  description?: string;
  /** What this option does to the staging area */
  stagingAction?: StagingAction;
}

// Phase builders — each returns the step queue for that phase
export function buildGetClearSteps(inboxItems: InboxItem[]): ReviewFlowStep[];
export function buildGetCurrentSteps(
  staleAtoms: Atom[],
  projectsMissingNextAction: SectionItem[],
  compressionCandidates: CompressionCandidate[],
): ReviewFlowStep[];
export async function buildGetCreativeSteps(
  sections: Section[],
  recentDecisions: Atom[],
  recentInsights: Atom[],
  signal?: AbortSignal,
): Promise<ReviewFlowStep[]>;

// Phase boundary: AI-generates a ~200 token summary of what happened in the phase
export async function generatePhaseSummary(
  phase: ReviewPhase,
  actionsTaken: ReviewAction[],
  signal?: AbortSignal,
): Promise<string>;
```

### Pattern 3: Staging Area (ephemeral module-level signals)
**What:** In-memory proposal queue that accumulates proposals from all three review phases. Not part of `BinderState` (avoids worker reconciliation issues). Cleared when review ends.
**When to use:** This is the ONLY correct place for pre-commit AI proposals.

```typescript
// In src/ui/signals/store.ts — following triageSuggestions pattern

export type StagingProposalType = 'new-atom' | 'mutation' | 'deletion';

export interface NewAtomProposal {
  type: 'new-atom';
  id: string;                          // ephemeral ID (not a real atom ID)
  proposedAtom: Omit<CreateAtomInput, 'id' | 'created_at' | 'updated_at'>;
  reasoning: string;
  source: 'get-clear' | 'get-current' | 'get-creative';
}

export interface MutationProposal {
  type: 'mutation';
  id: string;                          // ephemeral proposal ID
  atomId: string;                      // real atom to mutate
  currentAtom: Atom;                   // snapshot for display (current state)
  proposedChanges: Partial<Atom>;      // what to change
  reasoning: string;
  source: 'compression-coach' | 'get-current';
}

export interface DeletionProposal {
  type: 'deletion';
  id: string;                          // ephemeral proposal ID
  atomId: string;
  atom: Atom;                          // snapshot for display
  proposedAction: 'archive' | 'delete';
  reasoning: string;
  source: 'compression-coach';
}

export type StagingProposal = NewAtomProposal | MutationProposal | DeletionProposal;

// Module-level (NOT in BinderState — ephemeral)
const [stagingProposals, setStagingProposals] = createSignal<StagingProposal[]>([]);
export { stagingProposals };

// Staging operations
export function addStagingProposal(proposal: StagingProposal): void;
export function removeStagingProposal(proposalId: string): void;
export function clearStagingArea(): void;

// Commit approved proposal — goes through normal mutation pipeline
export function approveProposal(proposalId: string): void;

// Approve all
export function approveAllProposals(): void;
```

### Pattern 4: Compression Coach Engine
**What:** A pure module that takes `compressionCandidates` + full `atoms` + `scores` and produces per-candidate AI explanations referencing specific signals. Cloud API for nuanced reasoning.
**When to use:** During Get Current phase and (future) standalone orb entry.

```typescript
// src/ai/compression.ts — new module
export interface CompressionExplanation {
  atomId: string;
  title: string;
  explanation: string;       // AI-written, references specific signals
  staleDays: number;
  linkCount: number;
  similarAtomCount: number;  // atoms with keyword similarity > 0.3
  decisionContext?: string;  // relevant decision atom title if found
  recommendedAction: 'archive' | 'delete' | 'tag-someday' | 'add-link';
  confidence: 'high' | 'medium' | 'low';
}

export async function generateCompressionExplanations(
  candidates: CompressionCandidate[],
  atoms: Atom[],
  scores: Record<string, AtomScore>,
  signal?: AbortSignal,
  onProgress?: (count: number, total: number) => void,
): Promise<CompressionExplanation[]>;

// Prompt pattern: one API call per candidate (or batched)
// Each call receives: atom content + staleness days + link count + similar atom titles
// Returns: explanation + recommended action
```

**Batching strategy:** The compression coach generates one AI explanation per candidate. With cloud API and pre-send approval per call, multiple candidates means multiple approval modals — this is a UX problem. **Resolution:** Batch all candidates into a single cloud call that returns a JSON array of explanations. Reduces to one approval modal per compression coaching session.

```typescript
// Batch prompt structure for compression:
const batchPrompt = `You are analyzing a knowledge management system for compression candidates.
For each item below, explain in ONE sentence why it may be noise (stale, redundant, or superseded).
Reference specific signals: staleness days, how many similar items exist, whether any decisions are related.

CANDIDATES:
${candidates.map((c, i) => `
${i + 1}. Title: "${atom.title}"
   Staleness: ${staleDays} days since last update
   Link count: ${atom.links.length} (orphaned = 0)
   Similar atoms: ${similarTitles.join(', ') || 'none'}
   Related decisions: ${relatedDecisions.join(', ') || 'none'}
`).join('\n')}

Respond with ONLY a JSON array with objects containing:
{ "atomId": "...", "explanation": "...", "recommendedAction": "archive|delete|tag-someday|add-link", "confidence": "high|medium|low" }`;
```

### Pattern 6: Applying Approved Mutations with source: 'ai'
**What:** Approved staging proposals go through the EXACT same mutation pipeline as manual user edits, but with `source: 'ai'` in the changelog entry. The existing undo system already handles this correctly.
**When to use:** When user approves any proposal from the staging area.

```typescript
// Approving a MutationProposal:
sendCommand({
  type: 'UPDATE_ATOM',
  payload: {
    id: proposal.atomId,
    changes: {
      ...proposal.proposedChanges,
      // source field goes into changelog, not atom data
    },
    source: 'ai',            // NEW: needs to be threaded through UPDATE_ATOM command
    aiRequestId: proposal.id,
  },
});

// IMPORTANT: The messages.ts Command type and worker handler NEED UPDATE:
// UPDATE_ATOM payload must accept optional `source: 'ai' | 'user'` and `aiRequestId`
// The handler passes these to appendMutation() via the MutationLogEntry
```

**Critical path for AIGN-04:** The `appendMutation()` function in `changelog.ts` creates `MutationLogEntry` objects, which already have `source` and `aiRequestId` fields in the Zod schema. However, the `handleUpdateAtom` handler in `worker/handlers/atoms.ts` calls `appendMutation(id, 'update', before, after)` without passing `source`. The handler needs a way to receive `source: 'ai'` from the command payload and forward it to the log entry.

### Anti-Patterns to Avoid
- **Open-ended chat in Get Creative:** Each Get Creative step must have a defined `stepId`, a bounded set of options, and a clear "I'm done with this step" path. Never allow the user to keep asking follow-up questions within a step.
- **Adding staging proposals to BinderState:** Worker reconcile via `reconcile()` should never touch staging data. Keep staging as pure module-level `createSignal`. SolidJS's fine-grained reactivity works correctly with module-level signals.
- **Multiple cloud approval modals per review:** Batch all cloud calls within a phase transition into a single call where possible. The CloudAdapter fires the pre-send approval handler once per `execute()` call. Plan: one call for phase summary, one call for compression batch, one call for Get Creative pattern surfacing.
- **Persisting staging proposals to Dexie:** If the user force-closes mid-staging-review, proposals are lost. This is CORRECT per the locked decision: "unapproved proposals discarded when session ends." Do NOT save staging to the review session.
- **Storing `source: 'ai'` on the Atom itself:** The `source` field belongs in `MutationLogEntry`, not in the Atom schema. Atoms don't need to know they were AI-modified; the changelog tracks this for undo purposes.
- **Implementing back-navigation within review flow:** The GTD review is a "get through it" flow, not an editable form. Steps do not support back navigation (unlike GTDAnalysisFlow which has a Back button). Each step advances forward only.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Compression signal aggregation | Custom similarity metric | `findRelatedAtoms()` from `src/ai/similarity.ts` | Jaccard keyword overlap already implemented, tested, and sufficient for identifying similar atoms to reference in explanations |
| Cloud batch approval | Per-call approval modal for each candidate | Single batched prompt + one approval per phase | CloudAdapter pre-send approval fires per `execute()`. Batching into one call = one modal. Design prompts accordingly. |
| Review step progress indicator | Custom progress component | Extend existing `review-briefing-progress` CSS class | Phase 6 already has spinner + progress message patterns; reuse the same CSS structure |
| AI mutation undo | Custom "undo AI change" button | Existing UNDO command in worker.ts | The UNDO handler reads `lastEntry.before` and restores; AI mutations already have `source: 'ai'` in `MutationLogEntry`; undo works identically for AI and user mutations |
| Staging area ID generation | UUID from atom-like ID | `crypto.randomUUID()` for ephemeral proposal ID | Proposals need unique IDs only for `removeStagingProposal()` lookup; ephemeral UUIDs are correct and simple |
| Review flow step count | Dynamic count based on items | Static step queue built at phase start | Build the full step queue when entering each phase. The queue length is known upfront (inbox items count, project count, etc.). Simple array, no dynamic recounting. |

**Key insight:** The staging area is not a database, not a Dexie table, not a new Atom type. It is a transient JavaScript array of proposal objects living in module-level signal state. Treating it as ephemeral (cleared on review end) is a feature, not a limitation.

## Common Pitfalls

### Pitfall 1: UPDATE_ATOM command doesn't thread source: 'ai' through to changelog
**What goes wrong:** AI-approved mutations appear in changelog without `source: 'ai'`, making them indistinguishable from user edits. AIGN-04 fails.
**Why it happens:** The `Command` type for `UPDATE_ATOM` currently accepts only `{ id: string; changes: Partial<Atom> }`. The worker handler calls `appendMutation()` without a `source` argument.
**How to avoid:**
1. Extend `UPDATE_ATOM` payload in `messages.ts` to accept optional `source?: 'user' | 'ai'` and `aiRequestId?: string`
2. Update `handleUpdateAtom` in `worker/handlers/atoms.ts` to receive and forward these to `appendMutation()`
3. Update `appendMutation()` in `changelog.ts` to accept optional `source` and `aiRequestId` parameters
4. `CREATE_ATOM` similarly needs `source?: 'ai'` threading for new atom proposals

**Warning signs:** `db.changelog.toArray()` showing entries with `source: undefined` after approving AI proposals.

### Pitfall 2: ReviewPhaseContext context window growth
**What goes wrong:** A long review with many items causes the prompt for each Get Creative step to exceed token limits as previous steps are naively concatenated.
**Why it happens:** Each API call naively includes all previous answers in the prompt.
**How to avoid:** Implement the locked summarization strategy:
- Within a phase: include only the current step data + phase summary (not all previous answers)
- At phase boundary: call AI once to summarize the completed phase into ~200 tokens
- Next phase: system prompt includes phase summaries (not full answer trails)
- This keeps every API call under ~2000 tokens regardless of review length
**Warning signs:** Cloud API returning 400 errors about token limits; UI hanging on steps with many prior answers.

### Pitfall 3: Staging area proposals referencing stale atom snapshots
**What goes wrong:** A `MutationProposal` was created with `currentAtom` snapshot. Between proposal creation and approval, the atom was modified by a worker STATE_UPDATE. The displayed "current → proposed" diff is stale.
**Why it happens:** Staging proposals store a snapshot of the atom at creation time. Worker STATE_UPDATE pushes fresh atom data. The snapshot and real atom diverge.
**How to avoid:** When approving a `MutationProposal`, re-read the current atom from `state.atoms` (not from `proposal.currentAtom`) before constructing the `UPDATE_ATOM` command. The displayed diff in the staging view can use the snapshot for presentation, but the actual mutation merges proposed changes into the CURRENT atom state.
**Warning signs:** Approving a tag-add that already exists; mutation overwriting a user's manual edit made after the proposal was generated.

### Pitfall 4: Multiple cloud approval modals during compression coaching
**What goes wrong:** User triggers "Start Review", enters Get Current, compression coach generates 5 candidates, and 5 cloud approval modals appear in sequence. User frustration.
**Why it happens:** CloudAdapter fires `onPreSendApproval` per `execute()` call. If `generateCompressionExplanations()` loops candidates and calls `dispatchAI()` per candidate, each triggers a modal.
**How to avoid:** Design `generateCompressionExplanations()` to send ONE batched prompt containing all candidates (see Pattern 4 above). Single `dispatchAI()` call = one modal for the entire compression coaching session.
**Warning signs:** Multiple `CloudRequestPreview` modals appearing during a single review step.

### Pitfall 5: Get Creative "area gap check" breaks when sections are empty
**What goes wrong:** Area gap check step asks about each area/section. If sections array is empty or has only `projects`, the step queue is empty and the phase skips silently.
**Why it happens:** `buildGetCreativeSteps()` iterates over `sections` to build area gap questions. Empty sections = no steps.
**How to avoid:** Add fallback: if no sections exist, show at least the trigger list and Someday/Maybe steps. The area gap check step is only added per non-projects section. Guard with `if (nonProjectSections.length > 0)` before adding those steps.
**Warning signs:** Get Creative phase completing in 1-2 steps instead of 5-8.

### Pitfall 6: ReviewSession shape mismatch breaks Phase 6 resume
**What goes wrong:** Phase 7 extends `ReviewSession` in `review-session.ts`. Existing Phase 6 sessions persisted to Dexie config table have the old shape. Loading them causes `ReviewPhaseContext` to be undefined.
**Why it happens:** `loadReviewSession()` casts the Dexie entry as `ReviewSession` with no migration.
**How to avoid:** Add optional fields to `ReviewSession` (not required fields) so old sessions still parse correctly. Check `session.reviewPhase ?? null` and `session.reviewPhaseContext ?? null` everywhere the new fields are accessed.
**Warning signs:** Phase 6 briefing view crashing on load for users who have a pre-Phase 7 session; TypeScript errors on `session.reviewPhase` being `undefined`.

### Pitfall 7: ConversationTurnCard vs AIQuestionFlow — component collision
**What goes wrong:** Building `ConversationTurnCard` as a wrapper around `AIQuestionFlow` (the modal). They have fundamentally different mounting patterns.
**Why it happens:** `AIQuestionFlow` uses a fixed-position backdrop overlay with `show/hide` signal. The review flow needs inline cards within a scrollable view.
**How to avoid:** Build `ConversationTurnCard` as an entirely separate component. Share only the `QuestionFlowOption` and `QuestionFlowContext` type interfaces (both need the same option shape). Do NOT reuse the `showQuestionFlow` signal — that controls the modal overlay, not the review flow.
**Warning signs:** Review flow steps appearing as modal overlays that block the review view itself.

const [stagingProposals, setStagingProposals] = createSignal<StagingProposal[]>([]);
export { stagingProposals };

export function addStagingProposal(proposal: StagingProposal): void {
  setStagingProposals((prev) => [...prev, proposal]);
}

export function removeStagingProposal(proposalId: string): void {
  setStagingProposals((prev) => prev.filter((p) => p.id !== proposalId));
}

export function clearStagingArea(): void {
  setStagingProposals([]);
}

export function approveProposal(proposalId: string): void {
  const proposals = stagingProposals();
  const proposal = proposals.find((p) => p.id === proposalId);
  if (!proposal) return;

  switch (proposal.type) {
    case 'new-atom':
      sendCommand({ type: 'CREATE_ATOM', payload: { ...proposal.proposedAtom, source: 'ai', aiRequestId: proposal.id } });
      break;
    case 'mutation':
      // Re-read current atom from store to avoid stale snapshot issue
      const currentAtom = state.atoms.find((a) => a.id === proposal.atomId);
      if (!currentAtom) return;
      sendCommand({
        type: 'UPDATE_ATOM',
        payload: { id: proposal.atomId, changes: proposal.proposedChanges, source: 'ai', aiRequestId: proposal.id },
      });
      break;
    case 'deletion':
      if (proposal.proposedAction === 'archive') {
        sendCommand({
          type: 'UPDATE_ATOM',
          payload: { id: proposal.atomId, changes: { status: 'archived' }, source: 'ai', aiRequestId: proposal.id },
        });
      } else {
        sendCommand({ type: 'DELETE_ATOM', payload: { id: proposal.atomId } }); // DELETE_ATOM needs source: 'ai' too
      }
      break;
  }

  removeStagingProposal(proposalId);
}

export function approveAllProposals(): void {
  for (const proposal of stagingProposals()) {
    approveProposal(proposal.id);
  }
}

const COMPRESSION_BATCH_SCHEMA = {

        type: 'object',
        properties: {
          atomId: { type: 'string' },

          recommendedAction: { type: 'string', enum: ['archive', 'delete', 'tag-someday', 'add-link'] },
          confidence: { type: 'string', enum: ['high', 'medium', 'low'] },
        },
        required: ['atomId', 'explanation', 'recommendedAction', 'confidence'],
      },
    },
  },
  required: ['explanations'],
};

export async function generateCompressionExplanations(
  candidates: CompressionCandidate[],
  atoms: Atom[],
  scores: Record<string, AtomScore>,
  signal?: AbortSignal,
): Promise<CompressionExplanation[]> {
  const enriched = candidates.map(c => {
    const atom = atoms.find(a => a.id === c.id);
    if (!atom) return null;
    const staleDays = Math.floor((Date.now() - atom.updated_at) / 86400000);
    const similarAtoms = findRelatedAtoms(
      atom.content + ' ' + atom.title,
      atoms.filter(a => a.id !== c.id).map(a => ({ id: a.id, title: a.title, content: a.content })),
      5,
    );
    return { atom, staleDays, linkCount: atom.links.length, similarAtomTitles: similarAtoms.map(id => atoms.find(a => a.id === id)?.title ?? id) };
  }).filter(Boolean);

  // Single batched prompt → one cloud approval modal
  const prompt = buildCompressionBatchPrompt(enriched);
  const response = await dispatchAI({ requestId: crypto.randomUUID(), prompt, maxTokens: 800, signal });
  return parseCompressionBatchResponse(response.text, enriched);
}

## Open Questions

1. **CREATE_ATOM command source field threading**
   - What we know: `UPDATE_ATOM` needs `source: 'ai'` threading. `CREATE_ATOM` also needs it for new atom proposals from Get Creative.
   - What's unclear: Whether `handleCreateAtom` signature also needs updating, or if `aiSourced: true` (already on some `CreateAtomInput` schemas) is sufficient.
   - Recommendation: Check `atoms.ts` — `BaseAtomFields` includes `aiSourced: z.boolean().optional()`. For the atom body, use `aiSourced: true`. For the changelog entry, thread `source: 'ai'` separately. Both needed: `aiSourced` appears on the atom for UI badge display; `source` in the changelog for audit trail.

2. **DELETE_ATOM source tracking**
   - What we know: Deletion proposals in staging area can approve as `DELETE_ATOM`. The current `DELETE_ATOM` command has no `source` field.
   - What's unclear: Whether deletion needs `source: 'ai'` in the changelog (AIGN-04 says "fully reversible via undo" — undo already works for deletes via before-snapshot; tagging is for auditing).
   - Recommendation: Also extend `DELETE_ATOM` payload with optional `source?: 'ai'` for AIGN-04 compliance. Low risk — purely additive.

3. **Review flow routing — new 'review-flow' page or reuse 'review' page**
   - What we know: `activePage === 'review'` currently routes to `ReviewBriefingView`. The guided review flow is a different view state.
   - What's unclear: Whether to add a new `'review-flow'` page constant or use a sub-state within the existing review route.
   - Recommendation: Add `'review-flow'` as a new `activePage` value. Clean separation: `'review'` = briefing, `'review-flow'` = guided. Add `Match when={state.activePage === 'review-flow'}` in `MainPane.tsx`. Routing is cheap.

4. **Phase summary AI call — local or cloud**
   - What we know: Phase summaries are generated at phase transitions (~200 tokens output). They're used as context for the next phase's questions.
   - What's unclear: Whether these summaries need cloud-quality reasoning or if WebLLM is sufficient.
   - Recommendation: Use the active adapter (via `dispatchAI()`). If cloud is active, cloud handles it; if local only, WebLLM does it. Phase summaries are structured and short — WebLLM should handle them. No special casing needed.

5. **Get Current phase with no stale items or missing next actions**
   - What we know: Get Current step queue is built from stale atoms + projects missing next actions. If the system is healthy (Phase 6 shows "great shape!"), this queue could be empty.
   - What's unclear: Whether Get Current phase skips entirely or shows a "all good" message and advances.
   - Recommendation: If the Get Current step queue is empty, show a single "Your projects are all healthy — no stale items or missing next actions" card, then auto-advance to Get Creative. Don't skip silently.
