  <name>Task 1: Add ONNX classifier loading with Cache API and progress reporting to embedding worker</name>
  <files>src/search/embedding-worker.ts</files>
  <action>
Add ONNX classifier infrastructure to the existing embedding worker. This adds alongside the existing MiniLM pipeline — the two coexist in the same worker.

**1. Import and configure ONNX Runtime Web at the top of the file (after the Transformers.js imports):**

```typescript
import * as ort from 'onnxruntime-web';

// Disable proxy mode (we're already inside a worker) and multi-threading
// (tiny model, avoids SharedArrayBuffer requirement — per RESEARCH.md Pitfall 6)
ort.env.wasm.proxy = false;
ort.env.wasm.numThreads = 1;
```

**2. Add classifier state variables (after the pipeline singleton section):**

```typescript
// --- ONNX Classifier state ---
const CLASSIFIER_CACHE_NAME = 'onnx-classifier-v1';
const CLASSIFIER_MODEL_PATH = 'models/classifiers/triage-type.onnx';
const CLASSIFIER_CLASSES_PATH = 'models/classifiers/triage-type-classes.json';

let classifierSession: ort.InferenceSession | null = null;
let classMap: Record<string, string> | null = null;  // {"0":"decision","1":"event",...}
let classifierLoading = false;
```

**3. Add `fetchWithCache` function (Cache API + download progress):**

Implement a function that:
- Opens the cache with `caches.open(CLASSIFIER_CACHE_NAME)`
- Checks for a cached response with `cache.match(url)`
- If cached, returns the ArrayBuffer directly (no progress events for cache hits)
- If not cached, fetches with `ReadableStream` reader for chunk-by-chunk progress
- Reports progress via `self.postMessage({ type: 'CLASSIFIER_PROGRESS', percent })` where percent is 0-100 or -1 if `content-length` header is missing
- After fetch completes, stores in cache via `cache.put(url, new Response(buffer, ...))`
- Returns the `ArrayBuffer`

**4. Add `cleanOldCaches` function:**

Delete any `caches` keys starting with `'onnx-classifier-'` that don't match `CLASSIFIER_CACHE_NAME`. This auto-cleans old model versions.

**5. Add `loadClassifier` function:**

This is called at worker startup (eager loading per locked decision). It:
- Guards against double-loading (`if (classifierLoading || classifierSession) return`)
- Sets `classifierLoading = true`
- Calls `cleanOldCaches()`
- Resolves the model URL relative to the worker's origin (handles both `/` dev and `/BinderOS/` production base paths). Use the pattern: determine base from `self.location` — if pathname starts with `/BinderOS/`, prefix paths with `/BinderOS/`; otherwise use `/`.
- Fetches `triage-type-classes.json` (small JSON, can use simple fetch + cache) and parses into `classMap`
- Fetches `triage-type.onnx` via `fetchWithCache` with progress reporting
- Creates `ort.InferenceSession.create(new Uint8Array(modelBuffer), { executionProviders: ['wasm'] })`
- Sets `classifierSession` on success
- Posts `{ type: 'CLASSIFIER_READY' }` on success
- On ANY error: sets `classifierLoading = false`, posts `{ type: 'CLASSIFIER_ERROR', error: message }`, and logs to console. Does NOT throw — graceful degradation per INFER-03.

**6. Add `runClassifierInference` function:**

Takes a `number[]` embedding (384-dim) and returns `Record<string, number>` (label -> probability). It:
- Throws if `classifierSession` or `classMap` is null
- Creates an input tensor: `new ort.Tensor('float32', Float32Array.from(embedding), [1, 384])`
- Runs inference: `session.run({ [session.inputNames[0]]: inputTensor })`
- Finds the probability output: look for output name containing 'prob', or fallback to `outputNames[1]` then `outputNames[0]`
- Maps probabilities to class labels using `classMap`
- Returns the scores record

**7. Add `CLASSIFY_ONNX` and `LOAD_CLASSIFIER` to the WorkerIncoming union:**

```typescript
| { type: 'CLASSIFY_ONNX'; id: string; text: string }
| { type: 'LOAD_CLASSIFIER' }
```

Note: `CLASSIFY_ONNX` receives TEXT (not embedding). The handler embeds the text using the existing `embedTexts()` function, then runs ONNX inference. This keeps the same external API as `CLASSIFY_TYPE` and avoids an extra round-trip.

**8. Add message handlers in `self.onmessage`:**

For `LOAD_CLASSIFIER`: call `loadClassifier()` (fire-and-forget, errors handled internally).

For `CLASSIFY_ONNX`:
- Try to embed the text via `embedTexts([msg.text])`
- Try to run `runClassifierInference(vector)` on the embedding
- Post `{ type: 'ONNX_RESULT', id: msg.id, scores, vector }` on success (includes vector for ClassificationEvent caching)
- Post `{ type: 'ONNX_ERROR', id: msg.id, error: message }` on any failure

**9. Call `loadClassifier()` at the bottom of the file** (eager loading at worker init, per locked decision). Wrap in void to avoid unhandled rejection: `void loadClassifier();`

**Important notes:**
- Do NOT use `ort.env.wasm.wasmPaths` — let ORT resolve WASM paths from `import.meta.url` which works correctly in Vite worker builds. Only add explicit wasmPaths if ORT fails to find its binaries (the research notes this as MEDIUM confidence).
- Keep all existing message handlers (`EMBED`, `EMBED_ATOMS`, `CLASSIFY_TYPE`, `ROUTE_SECTION`) completely unchanged — they still work for centroid-based classification and search.
- The `CLASSIFY_TYPE` handler remains for backward compat (used by section routing until Phase 12).
