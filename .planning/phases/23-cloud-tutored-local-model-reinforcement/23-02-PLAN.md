  <name>Task 1: Create adversarial data generator (51_generate_adversarial.py)</name>
  <files>scripts/train/51_generate_adversarial.py</files>
  <action>
Create `scripts/train/51_generate_adversarial.py` that generates deliberately hard/ambiguous examples near decision boundaries for each classifier via the Anthropic API.

**CLI interface:**
- `--classifier {name|all}` -- which classifier(s) to generate for (default: all)
- `--model {haiku|sonnet}` -- Claude model (default: haiku for bulk generation per user decision)
- `--count N` -- total examples to generate per classifier (default: 200; split proportionally favoring weakest classes)
- `--benchmark-file PATH` -- path to benchmark JSON from Plan 01 (default: auto-detect latest in reports/)
- `--dry-run` -- generate 5 examples per classifier and print to stdout without writing to JSONL

**Core logic:**

1. **Load benchmark results** from the JSON file to identify:
   - Weakest classes per classifier (lowest F1)
   - Low-confidence example patterns (what text confuses the model)
   - Overall accuracy (skip classifiers above 99% accuracy -- diminishing returns)

2. **Allocate generation budget per class** based on weakness:
   - Classes with F1 < 0.90: get 40% of the budget
   - Classes with F1 0.90-0.95: get 35%
   - Classes with F1 > 0.95: get 25%
   - For decomposition (35 classes): focus only on bottom-10 F1 classes

3. **Generate adversarial examples** using indirect prompts (CRITICAL: avoid label leakage -- see RESEARCH.md Pitfall 2):

   For each target class, create prompts that describe the SCENARIO, not the label:
   - "Generate a realistic GTD inbox item where someone [scenario description]. The text should be messy, short, and natural -- the way real people type quick notes to themselves."
   - Include GTD methodology context in every prompt: 2-minute rule, horizons of focus, natural planning model, someday/maybe boundaries, weekly review criteria, context-dependent next actions
   - Rotate through difficulty tiers: easy (clear but natural), medium (requires GTD knowledge), hard (boundary case between two classes), adversarial (deliberately ambiguous but has a defensible correct answer)

   For the TYPE classifier specifically, include the augmentation categories from CONTEXT.md:
   - Compound tasks: "Call dentist and then update insurance info", "Fix the link, then notify Sarah"
   - Ambiguous facts that look like tasks: "The faucet is leaking", "The homepage link is broken"
   - Vague events: "Dentist next week", "Meeting sometime Thursday afternoon"
   - Implicit decisions: "We're going with the new vendor", "Looks like we'll switch to AWS"
   - Frustrated-tone insights: "I always underestimate how long things take", "I keep forgetting to check the budget"

4. **Boundary pair generation**: For multi-class classifiers, generate examples that sit on the boundary between the TWO most confused classes (from the confusion matrix in benchmark results). Prompt: "Generate a GTD inbox item that could plausibly be EITHER [class_a] or [class_b], but is ultimately [correct_class] because [GTD reasoning]."

5. **Dedup and append**: Use case-insensitive exact text match to avoid duplicates with existing training data. Append to the existing JSONL files in scripts/training-data/ (augment, don't replace).

6. **Output summary**: Print per-classifier generation stats (N generated, N deduped, N appended) and total API cost estimate.

**Schema for structured output:**
```python
{
    "text": str,           # The GTD inbox item
    "label": str,          # enum of valid labels for this classifier
    "reasoning": str,      # WHY this label is correct (for quality verification)
    "difficulty": str,     # enum: easy, medium, hard, adversarial
}
```

Only write `text` and `label` to the JSONL (reasoning and difficulty are for generation quality only, not training data).

Rate limiting: 0.05s sleep between calls, exponential backoff on RateLimitError. Use tqdm progress bars for generation loops.

<task type="auto">
  <name>Task 2: Create gap analysis script (52_gap_analysis.py)</name>
  <files>scripts/train/52_gap_analysis.py</files>
  <action>
Create `scripts/train/52_gap_analysis.py` that has Claude systematically analyze what GTD methodology knowledge each classifier lacks.

**CLI interface:**
- `--classifier {name|all}` -- which classifier(s) to analyze (default: all)
- `--model {haiku|sonnet}` -- Claude model (default: sonnet for quality analysis per user decision)
- `--benchmark-file PATH` -- path to benchmark JSON (default: auto-detect latest in reports/)

**Core logic:**

1. **Load benchmark results** and the existing training data for each classifier.

2. **Sample existing training data** per classifier: randomly select 30-50 examples per class and send to Claude along with:
   - The classifier's class definitions (from registry gtd_definitions)
   - The confusion matrix and per-class F1 scores from the benchmark
   - The low-confidence examples from the benchmark

3. **Prompt Claude as a GTD methodology expert** (use Sonnet for quality):

   System prompt: "You are David Allen, creator of Getting Things Done (GTD). You have decades of experience coaching individuals and organizations in GTD implementation. You understand every nuance of the methodology: the 5 stages of mastering workflow (capture, clarify, organize, reflect, engage), the 6 horizons of focus, the 2-minute rule, the natural planning model, the someday/maybe distinction, context-based next actions, waiting-for tracking, weekly review criteria, and reference material organization."

   User prompt per classifier:
   "Analyze this GTD classifier's training data and performance. Identify SYSTEMATIC blind spots -- categories of GTD knowledge the model has never encountered.

   Classifier: {name}
   Classes: {class_names with definitions}
   Performance: {per_class F1 scores}
   Most confused pairs: {top-3 confused class pairs from confusion matrix}
   Low-confidence examples: {low_confidence_examples}

   Sample training data (30-50 examples per class):
   {sampled_examples}

   For each blind spot, explain:
   1. What GTD concept is missing (be specific -- e.g., 'horizons of focus level 3-5 items', not just 'advanced GTD')
   2. Why it matters for classification accuracy
   3. What kinds of examples would teach the model this concept
   4. 5 specific example texts with their correct labels that would fill this gap

   Focus on SYSTEMATIC gaps (patterns of missing knowledge), not individual misclassifications."

4. **Parse Claude's response** and structure into a Markdown report:

   ```markdown
   # Gap Analysis Report
   **Date:** {date}
   **Model:** {sonnet/haiku}

   ## Summary
   | Classifier | Gaps Found | Severity | Priority |
   |------------|-----------|----------|----------|

   ## Per-Classifier Analysis

   ### {classifier_name}
   **Overall accuracy:** {acc} | **Weakest class:** {class} ({f1})

   #### Gap 1: {gap_title}
   **GTD concept:** {specific concept}
   **Why it matters:** {explanation}
   **Missing examples:** {types of examples needed}

   **Suggested training examples:**
   | Text | Label | Difficulty |
   |------|-------|------------|
   ```

5. **Extract suggested examples** from the gap analysis and write them as JSONL to the existing training data files (same dedup logic as 51). This makes gap analysis a data source too, not just a report.

6. Write the full report to `scripts/train/reports/gap_analysis_{timestamp}.md`.

The gap analysis should identify systematic blind spots like:
- "Model has never seen horizons-of-focus reasoning (levels 3-5 items that aren't immediate next actions)"
- "Missing 2-minute rule boundary cases (items right at the 2-minute threshold)"
- "No examples of natural planning model outputs (brainstorming vs. organizing stages)"
- "Someday/maybe items that look like commitments but aren't"
- "Context-dependent actions that change based on available tools/location"
