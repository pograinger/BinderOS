  <name>Task 1: Create teacher-student distillation script (53_distill_labels.py)</name>
  <files>scripts/train/53_distill_labels.py</files>
  <action>
Create `scripts/train/53_distill_labels.py` that feeds low-confidence predictions to Claude Sonnet and gets expert labels with reasoning.

**CLI interface:**
- `--classifier {name|all}` -- which classifier(s) to distill for (default: all)
- `--model {haiku|sonnet}` -- Claude model (default: sonnet for quality distillation per user decision)
- `--count N` -- max examples to distill per classifier (default: 50)
- `--benchmark-file PATH` -- path to benchmark JSON (default: auto-detect latest)
- `--confidence-threshold F` -- below this confidence, examples are candidates for distillation (default: 0.80)

**Core logic:**

1. **Identify distillation candidates** from benchmark results:
   - Load the benchmark JSON
   - For each classifier, find examples where the model's confidence was below the threshold
   - These are examples the model is uncertain about -- prime candidates for teacher guidance
   - Also include examples where the model predicted incorrectly (regardless of confidence)

2. **Feed candidates to Claude Sonnet** with a teacher prompt:

   System prompt: "You are a GTD methodology expert and ML training data curator. Your job is to provide the CORRECT label for ambiguous GTD inbox items, along with detailed reasoning that explains the GTD principles behind the classification. Your reasoning should reference specific GTD concepts: the 5 stages of mastering workflow, 6 horizons of focus, 2-minute rule, natural planning model, someday/maybe criteria, waiting-for tracking, context lists, weekly review criteria."

   User prompt per example:
   "Classify this GTD inbox item for the {classifier_name} classifier.

   Text: '{example_text}'

   Available labels: {class_names_with_definitions}

   The current model predicted '{model_prediction}' with {confidence}% confidence.

   Provide:
   1. The CORRECT label (may agree or disagree with the model)
   2. Detailed GTD reasoning explaining WHY this label is correct
   3. Which GTD concepts are relevant to this classification
   4. A difficulty rating (easy/medium/hard/adversarial)"

3. **Process Claude's response:**
   - If Claude agrees with the model's prediction: this confirms the label, add the reasoning as a quality signal but don't necessarily add to training data (model already gets this right, just uncertain)
   - If Claude DISAGREES with the model's prediction: this is a correction -- add the example with Claude's label to the JSONL training data (these are the highest-value training signals)
   - Track agree/disagree statistics per classifier

4. **Structured output schema:**
   ```python
   {
       "label": str,            # Claude's expert label (enum of valid labels)
       "reasoning": str,        # Detailed GTD reasoning
       "gtd_concepts": [str],   # List of relevant GTD concepts
       "difficulty": str,       # easy/medium/hard/adversarial
       "agrees_with_model": bool # Whether Claude agrees with the original prediction
   }
   ```

5. **Append corrections to JSONL:** Only write examples where Claude disagrees with the model (these are the teacher's corrections). Write `text` and `label` (Claude's label) to the JSONL. Dedup against existing data.

6. **Write distillation report** to `scripts/train/reports/distillation_{timestamp}.md`:
   - Per-classifier summary: N candidates, N agreed, N corrected
   - Most common correction patterns (e.g., "model confuses waiting-for with someday-maybe")
   - GTD concepts most frequently cited in corrections
   - Example corrections with reasoning

Rate limiting: 0.05s sleep, exponential backoff. Use Sonnet by default (quality over quantity for distillation).

  <name>Task 2: Create retrain-and-report orchestrator (54_retrain_and_report.py)</name>
  <files>scripts/train/54_retrain_and_report.py</files>
  <action>
Create `scripts/train/54_retrain_and_report.py` that orchestrates the full retrain cycle and produces a before/after comparison report.

**CLI interface:**
- `--classifier {name|all}` -- which classifier(s) to retrain (default: all)
- `--before-benchmark PATH` -- path to the BEFORE benchmark JSON (default: auto-detect latest)
- `--skip-retrain` -- skip retraining, only re-benchmark and compare (useful for testing)
- `--expert-exam` -- also re-score against expert exam after retraining

**Core logic:**

1. **Record the "before" state:** Load the benchmark JSON from Plan 01 as the baseline.

2. **Retrain each classifier** by calling the existing train scripts via subprocess:
   - Type classifier: `python -u scripts/train/02_embed_data.py` THEN `python -u scripts/train/03_train_classifier.py` (CRITICAL: type classifier requires re-embedding because 02 writes embeddings_cache.npy that 03 reads -- see RESEARCH.md Pitfall 4)
   - GTD classifiers: `python -u scripts/train/21_train_gtd_classifier.py --classifier {name}` (self-contained, embeds from JSONL)
   - Decomposition: `python -u scripts/train/31_train_decomposition_classifier.py` (self-contained)
   - Clarification classifiers: `python -u scripts/train/41_train_clarification_classifier.py --classifier {name}` (self-contained)

   Run retraining SEQUENTIALLY (not parallel) -- GTD classifiers must run one at a time because single-threaded WASM backend errors on concurrent sessions (per STATE.md decision about ONNX sequential execution, which also applies to Python training with shared GPU/CPU resources).

   Capture stdout/stderr from each subprocess. Print progress. If any retrain fails, log the error and continue with remaining classifiers.

3. **Run Node.js validation** for each retrained classifier to confirm Python/Node parity:
   - Type: `node scripts/train/04_validate_model.mjs`
   - GTD: `node scripts/train/22_validate_gtd_models.mjs`
   - Decomposition: `node scripts/train/32_validate_decomposition_model.mjs`
   - Clarification: `node scripts/train/42_validate_clarification_models.mjs`

4. **Re-benchmark** by calling `python -u scripts/train/50_benchmark_models.py --classifier all` (and optionally `--expert-exam`) to get the "after" results.

5. **Produce before/after comparison report** at `scripts/train/reports/improvement_{timestamp}.md`:

   ```markdown
   # Classifier Improvement Report
   **Date:** {date}
   **Before benchmark:** {before_file}
   **After benchmark:** {after_file}

   ## Summary
   | Classifier | Before Acc | After Acc | Delta | Before Weakest F1 | After Weakest F1 | Delta |
   |------------|-----------|-----------|-------|-------------------|------------------|-------|

   ## Overall
   - Classifiers improved: N/12
   - Classifiers unchanged: N/12
   - Classifiers regressed: N/12 (ALERT if any)
   - Total augmented examples added: N

   ## Per-Classifier Details

   ### {classifier_name}
   **Status:** Improved / Unchanged / REGRESSED

   | Class | Before F1 | After F1 | Delta |
   |-------|----------|---------|-------|

   **Data augmentation:**
   - Adversarial examples added: N
   - Gap analysis examples added: N
   - Distillation corrections added: N
   - Total training examples: {before} -> {after}

   **Confidence distribution shift:**
   - Before: mean={mean}, median={median}, <threshold={pct}%
   - After: mean={mean}, median={median}, <threshold={pct}%

   **Examples of newly-correct predictions:**
   | Text | Before Pred (conf) | After Pred (conf) | Correct |
   |------|-------------------|-------------------|---------|

   ## Expert Exam Results (if --expert-exam)
   | Classifier | Before Exam Acc | After Exam Acc | Delta |
   ```

6. **Regression check:** If any classifier's overall accuracy dropped by more than 0.5%, print a prominent WARNING in the report and to stdout. This is the key safety gate -- cloud augmentation must not hurt existing performance.

7. **Print summary to stdout** with the most important numbers: total classifiers improved, any regressions, biggest improvement.

Note: This script does NOT modify existing train scripts. It calls them as subprocesses and reads their output. The 50_* numbering convention is extended to 54 for this orchestrator since it completes the pipeline.
