    - applyAnswer with skip records wasSkipped=true, advances to next question
    - computeGraduationReadiness returns maturity score and whether graduation should be offered
    - shouldReEvaluate(session, newContent) returns true if content changed significantly (>30% different from session start)
    - Session with no missing categories detected skips questions phase, goes straight to decompose-offer
  </behavior>
  <action>
    Create `src/ai/enrichment/enrichment-engine.ts` (PURE MODULE — no store imports):

    createEnrichmentSession(params: {
      inboxItemId: string;
      content: string;
      atomType?: AtomType;
      existingEnrichments?: Record<string, string>;
      missingCategories?: MissingInfoCategory[];
    }): EnrichmentSession
    - Parse content with parseEnrichment to detect existing enrichments
    - If missingCategories provided, use them; otherwise derive from what's NOT in existingEnrichments
    - Generate questions for missing categories using getQuestionsForCategories (imported from clarification/question-templates)
    - If no questions needed (all categories filled), start in 'decompose-offer' phase
    - Initialize provenance with OPERATION_IDS.ENRICH
    - Pre-fill answers from existingEnrichments

    advanceSession(session: EnrichmentSession, userChoice?: 'accept' | 'decline'): EnrichmentSession
    - Deterministic state transitions based on current phase and conditions
    - 'questions' -> 'decompose-offer' when currentQuestionIndex >= questions.length
    - 'decompose-offer' + accept -> 'decomposing'
    - 'decompose-offer' + decline -> 'graduate-offer'
    - 'decomposing' -> 'graduate-offer' when currentStepIndex >= decompositionSteps.length
    - 'graduate-offer' + accept -> 'graduating'
    - 'graduate-offer' + decline -> 'done'
    - 'graduating' -> 'done' (after graduation confirmed)

    applyAnswer(session: EnrichmentSession, answer: ClarificationAnswer): EnrichmentSession
    - Add answer to session.answers
    - Update provenance: add MISSING_INFO model if ONNX-derived question, add OPERATION_IDS.CLARIFY
    - Advance currentQuestionIndex
    - Return new session (immutable update pattern)

    applyDecompositionStep(session: EnrichmentSession, stepIndex: number, action: 'accept' | 'edit' | 'skip', editedText?: string): EnrichmentSession
    - Track accepted steps with type inference (default: 'task')
    - Update provenance: add DECOMPOSE_ONNX + OPERATION_IDS.DECOMPOSE
    - Advance currentStepIndex

    computeGraduationReadiness(session: EnrichmentSession): { maturityScore: number; shouldOffer: boolean }
    - Build enrichments record from session.answers
    - Compute maturity via computeMaturity()
    - shouldOffer if maturityScore >= 0.4 OR session has accepted decomposition steps

    shouldReEvaluate(session: EnrichmentSession, newContent: string): boolean
    - Compare newContent to session's original content
    - Return true if >30% character difference (simple Levenshtein ratio or length-based heuristic)
    - This enables the user decision: "smart re-evaluation: re-asks if atom content changed significantly"
  </action>

    buildGraduationProposal(session: EnrichmentSession, classificationContext?: {
      parentType?: AtomType;
      childSections?: Record<number, string>;
    }): GraduationProposal
    - Parent atom: use classificationContext.parentType or infer from enrichment (if has decomposition steps -> likely 'task', otherwise use ONNX type if available)
    - Parent content: original content + enrichment answers appended via appendEnrichment pattern
    - Parent quality: computeQuality with session provenance + maturity
    - Parent provenance: session.provenance | OPERATION_IDS.GRADUATE
    - Child atoms: from session.acceptedSteps, each with:
      - type: 'task' (default for decomposed steps)
      - content: step text
      - suggestedSection: from classificationContext.childSections or null
      - quality: computeQuality with step provenance + 0.6 maturity (decomposed steps are well-specified)
      - provenance: step provenance
      - included: true (default)

    toggleChildInclusion(proposal: GraduationProposal, childIndex: number): GraduationProposal
    - Immutable toggle of childAtoms[childIndex].included
    - Return new proposal

    getGraduationActions(proposal: GraduationProposal): Array<{
      action: 'classify-parent' | 'create-child';
      type: AtomType;
      content: string;
      sectionItemId?: string | null;
      provenance: number;
      skipTriage: boolean;
    }>
    - First entry: classify-parent (the inbox item becomes an atom)
    - Subsequent entries: create-child for each included child (skipTriage: true — per user decision "children skip re-triaging")
    - This returns descriptors, not actual store commands (pure module boundary)

    inferParentType(enrichments: Record<string, string>, hasDecomposition: boolean): AtomType
    - If decomposition accepted -> 'task' (multi-step = project/task)
    - If enrichment has "Next Action" -> 'task'
    - If enrichment has "Outcome" with decision language -> 'decision'
    - Default: 'task'
