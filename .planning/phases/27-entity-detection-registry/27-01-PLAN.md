    **Step 1: Create src/entity/types.ts.**
    - Define `EntityDetectionResult`: `{ mentions: EntityMention[]; newEntities: string[]; updatedEntities: string[] }`
    - Define `MatchResult`: `{ entityId: string; score: number }` for matcher output.
    - Define `RawNEREntity`: `{ text: string; type: string; start: number; end: number; confidence: number }` (reuse or re-export from sanitizer).
    - Export `ENTITY_CONFIDENCE_THRESHOLD = 0.7` as constant.
    - Export `AUTO_MERGE_THRESHOLD = 0.9` and `MERGE_CANDIDATE_THRESHOLD = 0.7` for dedup tiers.

    **Step 2: Create src/entity/entity-matcher.ts.**
    - Implement `EntityMatcher` interface with `normalize(text: string): string` and `matchScore(candidateName: string, existingEntity: { canonicalName: string; aliases: string[] }): number`.
    - `personMatcher`: normalize with NFC, strip titles (Dr., Mr., Mrs., Ms., Prof., Sir, Dame), lowercase, trim. Match: exact normalized = 1.0, alias exact = 1.0, substring inclusion (e.g. "Sarah" in "Sarah Chen") = 0.8.
    - `locationMatcher`: normalize with NFC, lowercase, trim. Match: exact = 1.0, alias = 1.0, no substring matching (prevents "New" matching "New York").
    - `orgMatcher`: normalize with NFC, strip suffixes (Inc., Ltd., Corp., LLC, Co., PLC), lowercase, trim. Match: exact = 1.0, alias = 1.0.
    - `getMatcherForType(type: 'PER' | 'LOC' | 'ORG'): EntityMatcher` switch function.

    **Step 3: Expand src/storage/entity-helpers.ts from stubs to full dedup.**
    - Add `findOrCreateEntity(text: string, type: 'PER' | 'LOC' | 'ORG'): Promise<string>` -- the main dedup entry point.
    - Logic: get matcher for type, load all entities of same type from Dexie (`db.entities.where('type').equals(type).toArray()`), run matchScore against each, pick best match.
    - If best score >= AUTO_MERGE_THRESHOLD (0.9): return existing entity ID, add text as new alias if not already present, increment mentionCount, update lastSeen.
    - If best score >= MERGE_CANDIDATE_THRESHOLD (0.7) but < 0.9: still link to existing entity (conservative auto-merge for Phase 27), add alias. Phase 29 UX will refine merge suggestions.
    - If best score < 0.7: create new entity via `createEntity()`.
    - When updating mentionCount/aliases/lastSeen on existing entity, use `db.entities.update(id, changes)`.
    - Add `decrementEntityMentionCount(entityId: string): Promise<void>` -- decrement mentionCount, keeping minimum 0.
    - Add `cleanupEntityMentionsForAtom(atomId: string): Promise<void>` -- read atomIntelligence, for each mention with entityId, decrement mentionCount. Then delete atomIntelligence row.

    **Step 4: Create src/entity/entity-detector.ts -- the detection orchestrator.**
    Pure module, no store imports. Orchestrates: NER call -> confidence filter -> registry lookup -> sidecar write.
    - Export `detectEntitiesForAtom(atomId: string, content: string): Promise<void>`.
    - Call `detectEntitiesForKnowledgeGraph(content)` from sanitizer.ts to get raw NER + DATE entities.
    - For each entity with type PER/LOC/ORG: call `findOrCreateEntity(text, type)` to get entityId.
    - For each entity with type MISC/DATE: entityId = undefined.
    - Build `EntityMention[]` array with all fields (entityText, entityType, spanStart, spanEnd, confidence, entityId).
    - Call `writeEntityMentions(atomId, mentions)` to persist to sidecar.
    - On edit (re-scan): first call `cleanupEntityMentionsForAtom` (decrement old counts), then run detection fresh. Use a flag or check if sidecar already has mentions.
    - IMPORTANT: Handle the re-scan case properly to avoid mention count drift (Pitfall 3). Before writing new mentions, read existing mentions and decrement counts for any that had entityId. Then write new mentions and the findOrCreateEntity calls handle incrementing.
    - Wrap everything in try/catch -- entity detection failures must NOT block atom operations.

    **Step 5: Wire lifecycle hooks in store.ts.**
    - Import `detectEntitiesForAtom` from `src/entity/entity-detector.ts`.
    - Import `cleanupEntityMentionsForAtom` from `src/storage/entity-helpers.ts`.
    - Find the CREATE_ATOM success handler (where sendCommand CREATE_ATOM resolves) and add: `void detectEntitiesForAtom(atomId, content)` as async fire-and-forget after successful save. The `void` operator makes clear it's intentionally non-awaited.
    - Find the UPDATE_ATOM success handler and add same: `void detectEntitiesForAtom(atomId, content)` after successful save. Detection orchestrator handles the re-scan (decrement old + detect new).
    - Find atom delete handling and add: `void cleanupEntityMentionsForAtom(atomId)` for cleanup.
    - IMPORTANT: These are fire-and-forget calls. Entity detection must NEVER block the atom save/update/delete flow. Use `void` prefix.
    - NOTE: Lifecycle hooks may need to be in store.ts (where CREATE_ATOM/UPDATE_ATOM commands are dispatched) OR in the worker handlers (src/worker/handlers/atoms.ts). Check which location has access to the content text after save and place the hooks there. If in worker, import directly. If in store.ts, the content may need to be extracted from the command payload. Prefer store.ts since entity-detector.ts imports from sanitizer.ts which manages the worker on the main thread.
