  <name>Task 1: Consensus worker and specialist runner with cold-start guard</name>
  <files>src/workers/consensus-worker.ts, src/ai/consensus/specialist-runner.ts, src/ai/consensus/index.ts</files>
  <action>
    Create `src/workers/consensus-worker.ts` — a Web Worker that:
    1. Imports `onnxruntime-web` and sets `ort.env.wasm.proxy = false; ort.env.wasm.numThreads = 1;` (same as embedding-worker.ts).
    2. On first `RUN_SPECIALISTS` message, lazy-loads 4 ONNX sessions from `/models/specialists/{name}-risk.onnx` paths. Stores them in a Map keyed by specialist name. Does NOT load on worker init — only on first request (lazy to avoid OOM on mobile before cold-start threshold is met).
    3. Accepts messages: `{ type: 'RUN_SPECIALISTS', id: string, fullVector: number[] }` where fullVector is the 84-dim flat canonical vector (task+person+calendar concatenated).
    4. For each specialist, extracts the feature slice using the specialist's featureIndices (passed in the message or hardcoded from SPECIALIST_FEATURE_SLICES — prefer passing from the runner to keep the worker generic). Runs `session.run({ X: tensor })` where tensor is `new ort.Tensor('float32', slicedFeatures, [1, slicedFeatures.length])`. Extracts probability of class 1 from the output (index [1] of the probabilities output tensor — shape [1, 2] with zipmap=False).
    5. Returns: `{ type: 'SPECIALIST_RESULTS', id: string, results: Array<{ name: string, probability: number }> }` or `{ type: 'SPECIALIST_ERROR', id: string, error: string }`.

    **Design decision:** Pass `featureSlices` configuration in the first `RUN_SPECIALISTS` message (or a separate `INIT` message). This keeps the worker generic and avoids importing TypeScript modules (workers can't import from src/ai/ easily). The runner sends featureSlices once; the worker caches them.

    Actually — simpler approach: the worker receives the full 84-dim vector + a `slices` array of `{ name: string, indices: number[] }`. The worker slices on each call. No INIT message needed; stateless except for cached sessions.

    Create `src/ai/consensus/specialist-runner.ts` — main-thread bridge:
    1. Module-level lazy worker instantiation: `let worker: Worker | null = null;` — created on first call to `runConsensusForAtom()`.
    2. Uses the requestMap pattern (Map<string, { resolve, reject }>) for request/response round-trips with the worker.
    3. `runConsensusForAtom(atomId: string, binderId: string)`: async function that:
       a. Checks cold-start guard: queries `db.atomIntelligence` for the binder's atoms that have `canonicalVector` with `vectorType === 'task'`. If count < 15, return early (do nothing). Use an in-memory counter (`vectorCountCache: Map<string, number>`) incremented on each call, initialized lazily with a Dexie count query on first call per binder. This avoids O(n) query on every invocation.
       b. Loads the atom's `atomIntelligence` row. If `canonicalVector` is undefined or `vectorType !== 'task'`, return early.
       c. Builds the full 84-dim vector. The stored canonical vector is only the task portion (27 dims). For consensus, we need task+person+calendar. However, not every task atom has an associated person or calendar. **Approach:** Use zero-padded person and calendar dims. The specialist models were trained on the full 84-dim vector, so missing person/calendar data = zeros. Concatenate: `[...taskVector.data, ...new Array(PERSON_VECTOR_DIM).fill(0), ...new Array(CALENDAR_VECTOR_DIM).fill(0)]`.
       d. Sends `RUN_SPECIALISTS` to worker with the full vector and SPECIALIST_FEATURE_SLICES config (from types.ts).
       e. On response, maps specialist results to `SpecialistOutput[]` (adding weights from `SPECIALIST_WEIGHTS`).
       f. Calls `computeConsensus(outputs)` to get the `ConsensusResult`.
       g. Calls `writeConsensusRisk(atomId, result)` — fire-and-forget.
    4. Export `runConsensusForAtom` and `incrementVectorCount` (for the vector-cache to call when it writes a new vector).

    Update `src/ai/consensus/index.ts` barrel to export `runConsensusForAtom`.

    After the existing `await db.atomIntelligence.put(intel);` succeeds, add:
    ```typescript
    // Fire-and-forget consensus computation for task atoms
    if (vectorType === 'task') {
      import('../consensus/specialist-runner').then(({ runConsensusForAtom, incrementVectorCount }) => {
        // Increment the in-memory vector count for this binder's cold-start tracking
        // Extract binderId from atomId (format: "binderId:atomUUID" or just the atomId if no colon)
        const binderId = atomId.includes(':') ? atomId.split(':')[0] : 'default';
        incrementVectorCount(binderId);
        void runConsensusForAtom(atomId, binderId);
      }).catch(() => {});
    }
    ```

    Note: This uses the same dynamic import pattern as the rest of vector-cache.ts. The consensus module is lazy-loaded only when a task vector is written. The `void` prefix on `runConsensusForAtom` makes the fire-and-forget intent explicit. The `.catch(() => {})` swallows import errors (e.g., if consensus module isn't available yet during testing).

    **Important:** The `binderId` extraction from `atomId` needs to match how atoms are stored. Check the actual atom ID format. If atoms don't have a binder prefix in their ID, the runner will need to look up the binderId from the atom record. Adjust accordingly — if atoms have a `binderId` field, pass it through as a parameter. The vector-cache callers (store.ts) already have the binderId in scope from the atom's parent binder.

    Actually, looking at store.ts wiring from Phase 35: the callers do `recomputeAndCacheVector(atom, sidecar, entities, relations)` where `atom` has a `binderId` field. But `writeCanonicalVector` only receives `atomId`. Two options:
    1. Add `binderId` parameter to `writeCanonicalVector` — but that changes the existing API.
    2. Have the consensus runner look up binderId from the atom record.

    Best approach: Add an optional `binderId` parameter to `writeCanonicalVector()`:
    ```typescript
    export function writeCanonicalVector(
      atomId: string,

      vector: Float32Array,
      binderId?: string,  // Phase 36: for consensus cold-start guard
    ): void {
    ```
    Then update `recomputeAndCacheVector` to pass `atom.binderId` through. This is a backward-compatible change (optional param).

    Update `recomputeAndCacheVector()` to pass binderId:

    ```typescript
    const result = computeTaskVector(atom as TaskAtom, sidecar, entities, relations);
    writeCanonicalVector(atom.id, 'task', result, (atom as any).binderId);
    ```

    If binderId is undefined in writeCanonicalVector, fall back to 'default' for the cold-start guard.
